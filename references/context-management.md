# 上下文管理协议

## 目标

让六路研究像六个独立研究员一样工作：各自拥有私有上下文，完成后用文件交接，只有在 Review 阶段才合并。

## 三类上下文

1. **Global Brief**：所有研究流共享的极小上下文，只包含公司身份、研究范围、时间边界、输出目录和用户特别要求。
2. **Stream Context**：每一路私有上下文，只包含本路任务、搜索过程、来源材料、判断和不确定性。
3. **Review Context**：六路都确认后生成的共享上下文，只读取各路交接卡、认知文件和必要原始证据。

## 隔离规则

1. 启动六路研究时，每路只接收 Global Brief 和本路任务说明。
2. 不把其他路完整 raw 文件、搜索过程或长对话历史传给当前路。
3. 如某一路需要其他路验证，只在 `需要其他路验证的问题` 中登记，不直接跨路合并。
4. 用户要求某一路返工时，只重开该路 Stream Context，其他已确认流保持不变。
5. Review 阶段只能在六路都达到用户确认状态后开始。

## 每路输出文件

每一路至少输出：

```text
[company]_[stream].md
[company]_[stream]_cognitive.md
[company]_[stream]_context_card.md
```

其中 `context_card` 是跨阶段交接的主入口，建议结构：

```markdown
# [公司] [研究流] Context Card

**状态**：[待确认 / 已确认 / 需返工]
**最后更新**：[YYYY-MM-DD HH:MM]
**负责人角色**：[A 财务 / B 业务 / C 战略 / D 领导思维 / E 批判视角 / F 文化符号]

## 已确认结论
- ...

## 关键证据
| 判断 | 来源 | 类型 | 可信度 |
|---|---|---|---|
| ... | URL 或文件位置 | 一手/二手 | high/medium/low |

## 不确定性
- ...

## 反例与冲突
- ...

## 需要其他路验证的问题
- ...

## 给 Review 阶段的压缩摘要
用 5-8 条 bullet 总结本路最重要的信息，避免粘贴完整搜索过程。
```

## Token 使用原则

1. 事实、数字、日期、URL 和原文短摘优先保留。
2. 搜索过程、重复背景、泛泛评价优先丢弃。
3. 每路交接卡控制在可快速阅读的长度；复杂证据放 raw 文件，交接卡只放索引。
4. Review 阶段优先读取 context_card，再读 cognitive；只有证据不足时回看 raw。
5. 最终 Skill 只保留通过验证的认知模型，不携带研究过程噪音。
